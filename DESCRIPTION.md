# Описание алгоритма поиска аномальных респондентов

## 1. Единица анализа и формула daily_ots

Алгоритм работает на уровне кортежей:

    (SubjectID, BrandID, CategoryNameDelivery, researchdate)

Для каждого кортежа вычисляется daily_ots по формуле из условия задачи:

    daily_ots(i, j, k) = Weight(i, k) × count_rows(i, j, k)

где count_rows — количество строк с BrandinDelivery = 1 и непустым
CategoryNameDelivery для данного SubjectID, BrandID, CategoryNameDelivery
и researchdate. Weight берётся как первое значение по (SubjectID, researchdate),
поскольку внутри одного дня у респондента один дневной вес.

Единица удаления — (SubjectID, researchdate): если хотя бы один бренд
дал аномалию, из выборки удаляются все строки этого респондента за этот день.

В коде: функция compute_daily_ots().

## 2. Почему нельзя применить обычный Z-score к raw daily_ots

Обычный Z-score z = (x − mean) / std предполагает нормальное распределение.

Проверка тестом Шапиро-Уилка выполняется на случайной подвыборке n = 5 000.
Ограничение n = 5 000 — это максимум, поддерживаемый реализацией в scipy;
полный датасет содержит более 50 000 строк.

Результат теста на raw daily_ots:
    stat = 0.4438, p = 1.81e-82
    → нормальность отвергается, вероятность ошибки пренебрежимо мала

Описательная статистика подтверждает сильную скошенность:
    skew     ≈ 17.6  (при нормальном распределении ≈ 0)
    kurtosis ≈ 836   (при нормальном распределении ≈ 0)

Физическая причина: daily_ots = Weight × count_rows. Оба множителя
правосторонне скошены. Weight панели может различаться на два порядка;
count_rows имеет медиану 1 и редкие значения до нескольких десятков.

Дополнительная проблема: у обычного Z-score breakdown point = 0%.
Один экстремальный выброс завышает mean и std, что занижает z-score
у остальных аномалий — они становятся невидимы. Это называется
эффектом маскировки выбросов.

## 3. Логарифмическое преобразование log(1 + daily_ots)

Применяется преобразование log(1 + x). Единица в аргументе защищает
от log(0) при гипотетически нулевом daily_ots.

Результат теста Шапиро-Уилка на той же подвыборке n = 5 000:
    stat = 0.9855, p = 2.71e-22

p < 0.05, но при n = 5 000 тест Шапиро-Уилка крайне чувствителен
и отвергает нормальность при любом отклонении. Содержательные показатели:
    skew     ≈ 0.45  (было ≈ 17.6)
    kurtosis ≈ 0.30  (было ≈ 836)

Принятый в прикладной статистике критерий допустимой асимметрии для
параметрических методов: |skew| < 1.0 (Hair et al., 2019).
После логарифма условие выполнено: 0.45 < 1.0.

В коде: строка daily["log_ots"] = np.log1p(daily["daily_ots"])
в функции compute_modified_zscore().

## 4. Modified Z-score (Iglewicz & Hoaglin, 1993)

Формула:

    M_i = 0.6745 × (x_i − median(X)) / MAD(X)

где x_i = log(1 + daily_ots_i), а
MAD = median(|x_i − median(X)|) — медианное абсолютное отклонение.

### Откуда константа 0.6745

0.6745 = Φ⁻¹(0.75) — 75-й перцентиль стандартного нормального
распределения. Это математический факт, не подбираемый параметр.

Для X ~ N(0, 1):
    median(X) = 0
    MAD = median(|X|) = Φ⁻¹(0.75) ≈ 0.6745

Следовательно: MAD = 0.6745 × std при нормальном распределении.

Симуляция (n = 10 000 000):
    Эмпирический MAD для N(0,1): 0.674227
    Теоретический Φ⁻¹(0.75):     0.674490

Умножение на 0.6745 нормирует MAD до эквивалента std, чтобы при
нормальных данных M_i совпадал с обычным z-score. Без этой константы
Modified Z-score был бы в 1/0.6745 ≈ 1.48 раза меньше обычного —
порог 3.5 ловил бы другой процент наблюдений.

### Преимущества перед обычным Z-score

Медиана и MAD имеют breakdown point = 50%: чтобы сдвинуть медиану,
нужно заменить более половины наблюдений. У обычного Z-score
breakdown point = 0%. Это ключевое свойство при работе с данными,
содержащими выбросы.

В коде: функция compute_modified_zscore(). Константы MZ_THRESHOLD = 3.5
и MZ_CONST = 0.6745 вынесены в начало файла.

## 5. Уровень группировки: по CategoryNameDelivery

M_i вычисляется внутри каждой категории отдельно — респондент
сравнивается только с теми, кто искал в той же товарной категории.

### Почему не глобально

Глобальный уровень объединяет все категории в одно распределение.
Это приводит к тому, что аномалия в небольшой категории (например,
Дезодоранты) тонет в глобальном хвосте, сформированном крупными
категориями (Смартфоны, Крупная техника). Реально аномальные записи
получают M_i ниже порога и остаются незамеченными.

### Почему не по (CategoryNameDelivery × researchdate)

Этот уровень порождает более 1 300 групп. Анализ показывает, что
среди них есть группы с MAD = 0 (все респонденты дали одинаковый
log_ots в этот день) — деление на ноль. Кроме того, в однодневных
группах с малым числом наблюдений z-score нестабилен: одиночный
респондент с обычным daily_ots, попавший в тихий день категории,
получает аномально высокий M_i не из-за своей активности, а из-за
малости группы сравнения. Это противоречит смыслу задачи.

### Почему категориальный уровень корректен

Все категории в данных имеют достаточно наблюдений (минимум — более
100 строк) и MAD > 0 во всех группах. Деление на ноль исключено,
специальная обработка граничных случаев не требуется.

Медианы log_ots по категориям однородны (разброс около 0.11 в
логарифмической шкале), что подтверждает корректность раздельного
анализа: Weight панели не систематически отличается между категориями.

В коде: группировка по "CategoryNameDelivery" в функции
compute_modified_zscore() через groupby и agg.

## 6. Порог M_i > 3.5

3.5 — стандартное значение из оригинальной работы Iglewicz & Hoaglin
(1993). Его статистический смысл: при нормальном распределении

    P(|Z| > 3.5) ≈ 0.000465

то есть менее 0.05% наблюдений превышают этот порог случайно.
Ожидаемое число случайных ложных срабатываний из N наблюдений:
N × 0.000465. Если фактическое число аномалий существенно выше
этого ожидаемого значения — алгоритм выявляет реальные аномалии,
а не шум.

Порог выбран из литературы и не подбирался под желаемый результат.

В коде: константа MZ_THRESHOLD = 3.5, условие в функции
detect_anomalies(): daily[daily["mz_score"] > MZ_THRESHOLD].

## 7. Что алгоритм гарантированно не удаляет

Малый OTS: M_i < 0 у всех записей с daily_ots ниже медианы своей
категории. Условие M_i > 3.5 физически недостижимо при маленьком
daily_ots.

Тяжёлый, но нормально активный респондент: высокий Weight без
аномального числа запросов даёт daily_ots в пределах нормы категории
и M_i < 3.5.

Единственный респондент бренда за конкретный день: z-score считается
по всей категории, а не внутри бренд-дня. Этот респондент конкурирует
с тысячами других записей категории, а не только с собой.

---

## 8. Детерминированность

Алгоритм не содержит случайных элементов: нет random, shuffle,
sample с непостоянным seed. Медиана, MAD и z-score — детерминированные
операции. При повторном запуске на тех же данных anomalies.csv
будет побайтово идентичен предыдущему.
